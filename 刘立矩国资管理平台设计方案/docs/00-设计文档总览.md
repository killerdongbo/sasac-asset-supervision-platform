## 2. 设计原则

1. **先文档，后开发**：开发前必须明确模块边界、数据结构、接口输入输出。
2. **巡检 + 盘点优先**：资产登记、分类、位置、二维码等基础能力都服务于巡检和盘点闭环。
3. **演示可用优先**：一期面向客户试用/演示，优先保证主流程可跑通、界面一致、数据链路完整。
4. **权限先行**：所有业务模块基于 RBAC 控制菜单、按钮和接口权限。
5. **资产卡片统一入口**：固定资产、无形资产、软件资产统一从资产卡片进入，但保留扩展属性。
6. **操作留痕**：资产状态变动、盘点差异调整、审批动作必须记录日志。
7. **可分阶段交付**：避免一次性开发全部 185+ 原型页面，按 P0/P1/P2/P3 优先级推进。

## 3. 交付阶段

### Phase 1：基础底座
- 登录认证
- 用户/角色/权限/部门
- 资产分类
- 资产位置
- 资产登记
- 资产清单
- 资产卡片基础信息

### Phase 2：核心业务
- 巡检任务
- 巡检执行
- 巡检异常处理
- 盘点任务
- 盘点执行
- 盘点差异报告
- 差异审批与调整

### Phase 3：业务扩展
- 采购申请、订单、验收
- 资产领用、派发、调拨、借用、归还、退库
- 维保、维修
- 审批流引擎

### Phase 4：财务与报表
- 折旧规则
- 折旧计提
- 统计报表
- 预警中心
- 条码/二维码打印
- 导入导出

## 4. 当前开发边界建议

建议下一步不要继续零散编码，而是按以下顺序执行：

1. 以 `03-数据库设计文档.md` 为准，整理 SQLAlchemy 模型。
2. 以 `04-API接口设计文档.md` 为准，整理 FastAPI 路由。
3. 以 `05-各功能开发指引文档.md` 为准，先完成 Phase 1。
4. Phase 1 验收通过后，再进入巡检/盘点 Phase 2。
