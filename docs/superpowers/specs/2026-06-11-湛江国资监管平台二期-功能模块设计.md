# 湛江国资国企一体化数字平台（二期）功能模块设计

> 2026-06-11 | 头脑风暴输出 | 状态：待审核

---

## 一、项目背景与目标

### 1.1 来源

基于 2026-06-11 湛江市国资国企一体化数字平台（二期）项目调研会议纪要，以及国资资产管理平台需求调研记录。

### 1.2 核心痛点

- **家底不清**：市国资委对下属企业国有资产无法全面掌握，数据依赖人工填报，质量差、时效性低
- **覆盖不足**：一期平台覆盖约11个集团，二期需扩展到约300-400家企业
- **上报困难**：省国资委下发19类报表任务，企业填报负担重、数据分散
- **合并报表水分**：集团合并报表存在内部交易未抵销，数据失真
- **信创要求**：省国资委新数据交换标准，平台需满足信创和等保要求

### 1.3 目标

- **2026年7月底**：交付可运行MVP，支持试点企业真实使用
- **核心聚焦**：固定资产全生命周期管理，逐步扩展到其他资产类型
- **长期定位**：从项目到产品，支持商业化推广

## 二、关键设计决策

| # | 决策项 | 选择 | 说明 |
|---|--------|------|------|
| 1 | 资产管理深度 | **全生命周期** | 从资产新增到处置报废的完整闭环 |
| 2 | 数据采集策略 | **混合模式** | Excel批量导入 + 财务系统对接 + 人工录入 |
| 3 | 组织架构层级 | **灵活多级树形** | 国资委→集团→子公司→孙公司，不限深度 |
| 4 | 数据上报机制 | **上报-确认模式** | 企业提交→集团审核→国资委接收，上报前自由修改，上报后走变更流程 |
| 5 | 财务对接深度 | **台账+科目余额+合并校验** | 拉取单体报表底层数据，平台侧做合并校验 |
| 6 | 部署模式 | **SaaS多租户云平台** | 统一部署，tenant_id数据隔离 |
| 7 | 7月交付物 | **功能可用MVP** | 支持试点企业实际使用和真实数据录入 |
| 8 | 系统架构 | **分层架构** | 接入层→应用层→业务服务层→数据层 |
| 9 | 技术栈 | **Java/SpringBoot全栈** | SpringBoot 3 + Vue3 + PostgreSQL，信创兼容 |

## 三、系统总体架构

### 3.1 分层视图

```
┌─────────────────────────────────────────────────────┐
│  接入层    │ Web管理端 │ 移动端/小程序 │ OpenAPI     │
├─────────────────────────────────────────────────────┤
│  应用层    │ 企业门户  │ 集团门户 │ 国资委驾驶舱 │ 系统管理 │
├─────────────────────────────────────────────────────┤
│  业务服务层 │ 资产登记 │ 变动管理 │ 折旧处置 │ 报表上报 │ 财务对接 │
├─────────────────────────────────────────────────────┤
│  数据层    │ 资产台账 │ 组织架构 │ 报表数据 │ 财务缓存 │ 配置字典 │
├─────────────────────────────────────────────────────┤
│  基础设施   │ PostgreSQL · Redis · MinIO · Nginx     │
└─────────────────────────────────────────────────────┘
```

### 3.2 MVP业务服务（5个）

| 服务 | 职责 |
|------|------|
| **资产登记服务** | 资产验收、入库、建卡、分类目录管理 |
| **变动管理服务** | 资产调拨、转让、抵押、闲置标记 |
| **折旧处置服务** | 折旧计提、报废、出售核销 |
| **报表上报服务** | 19类报表数据采集、审核链、上报、快照版本管理 |
| **财务对接服务** | 财务系统接口适配、数据拉取、清洗校验、合并抵销校验 |

### 3.3 MVP数据模型（5个）

| 模型 | 说明 |
|------|------|
| **Organization** | 组织架构树（id, parent_id, org_type, tenant_id） |
| **Asset** | 资产台账（核心主数据，包含分类、原值、净值、折旧方法、位置等） |
| **AssetChange / Depreciation / Disposal** | 资产生命周期事件记录 |
| **Report** | 报表数据（report_type, period, org_id, submit_status, version, snapshot_data） |
| **FinancialData** | 财务对接缓存（科目余额、固定资产卡片、单体报表） |

## 四、核心业务流程

### 4.1 资产全生命周期

```
资产新增 → 使用管理 → 变动流转 → 折旧摊销 → 资产处置
（采购验收、（领用·维修·（调拨·转让· （月折旧计提）（报废·出售·核销）
 划转入库）   盘点·闲置）  抵押·出租）             全程可追溯审计
```

### 4.2 数据上报-确认模式

| 阶段 | 角色 | 操作 | 约束 |
|------|------|------|------|
| **企业填报** | 企业操作员 | 系统自动汇总资产数据 → 企业补充校验 → 确认提交 | 提交前数据仅自己可见，可自由修改 |
| **集团审核** | 集团审核员 | 审核下属企业数据 → 可驳回修正 → 汇总生成集团报表 | 可穿透查看企业明细 |
| **国资委接收** | 国资委监管员 | 数据进入监管库 → 自动校验比对 → 生成监管报表 | 上报后锁定，变更需走变更流程 |
| **省平台对接** | 系统自动 | 按省数据标准转换 → 定期自动上报 → 接收反馈 | — |

**关键保护机制：**
- 缓冲空间：企业提交前数据仅自己可见，可反复修改
- 版本管理：每次上报生成快照版本，历史可查可对比
- 驳回重报：集团/国资委可驳回，企业修正后重新提交
- 变更流程：已上报数据修正走变更申请→审批→更新

### 4.3 财务系统对接

```
企业财务系统（用友/金蝶/SAP/其他）
    ↓
财务适配器层（标准API / 数据库直连 / Excel模板兜底）
    ↓
数据校验引擎（资产分类映射 / 合并冲抵校验 / 完整性检查）
    ↓
平台资产台账 & 财务数据缓存
```

**合并报表"水分"处理策略：**
不从企业直接拿合并报表数据，而是拉取底层单体报表数据 + 内部交易/往来标记，在平台侧做合并校验和抵销计算。

## 五、多租户与权限模型

### 5.1 数据隔离

| 维度 | 方案 |
|------|------|
| 数据库级 | 共享数据库 + tenant_id 字段隔离 |
| 应用级 | 请求上下文自动注入 tenant_id，数据访问层统一过滤 |
| 组织级 | 树形层级权限：上级可查看下级数据，同级隔离，跨分支不可见 |
| 角色级 | RBAC 四角色 |

### 5.2 角色定义

| 角色 | 权限范围 |
|------|---------|
| **国资委监管员** | 查看所有企业数据、驾驶舱、报表汇总，不可修改企业数据 |
| **集团审核员** | 查看本集团及下属企业、审核上报数据、汇总集团报表 |
| **企业操作员** | 管理本企业资产、填报数据、提交上报 |
| **系统管理员** | 租户管理、用户管理、配置字典、系统运维 |

## 六、技术选型

| 层次 | 选型 | 理由 |
|------|------|------|
| 前端框架 | Vue 3 + TypeScript + Element Plus | 政务项目主流选型，组件库成熟 |
| 可视化/驾驶舱 | ECharts 5 + DataV | 国产可视化库，大屏支持成熟 |
| 后端框架 | Spring Boot 3.x + MyBatis-Plus | 政务信创标配，生态完善 |
| 数据库 | PostgreSQL 15 + PostGIS | 开源、信创兼容，空间数据支持 |
| 缓存 | Redis 7 | Session共享、租户配置缓存 |
| 文件存储 | MinIO | 资产照片、附件、报表导出 |
| 报表引擎 | JasperReports + Apache POI | 复杂报表模板 + Excel导入导出 |
| 消息队列 | RabbitMQ（MVP可选） | 异步任务：批量折旧、报表生成 |
| API网关 | Spring Cloud Gateway | 统一鉴权、租户注入、限流 |
| 容器化 | Docker + Docker Compose | MVP够用，后续升级K8s |
| 信创适配 | 达梦/人大金仓 + 麒麟OS + 东方通 | 基于PG协议兼容，迁移成本低 |

## 七、部署架构（MVP）

```
Nginx (反向代理 + 静态资源 + HTTPS)
  ├─ Spring Cloud Gateway (鉴权·限流·租户注入)
  ├─ 前端 SPA (Vue3 Nginx静态)
  ├─ 资产服务 :8081 (SpringBoot)
  ├─ 报表服务 :8082 (SpringBoot)
  ├─ 财务对接服务 :8083 (SpringBoot)
  ├─ PostgreSQL :5432
  ├─ Redis :6379
  └─ MinIO :9000

全部 Docker Compose 编排 · 单机4C16G起步 · 后续可扩展K8s
```

## 十、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 7周交付压力大 | 质量不达标 | Sprint 3预留缓冲，范围可裁剪（报表先做3-5类） |
| 财务系统对接难度高 | 数据拿不到 | 适配器层设计灵活，Excel兜底，先对接1家再扩展 |
| 企业配合度不足 | 数据进不来 | Excel批量导入降低门槛，提供驻场培训支持 |
| 信创环境兼容 | 部署受阻 | 技术栈全部基于信创兼容选型，MVP可用标准环境先跑 |
