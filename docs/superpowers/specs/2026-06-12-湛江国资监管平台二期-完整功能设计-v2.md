## 一、修订说明

本文档在 [v1 设计文档](./2026-06-11-湛江国资监管平台二期-功能模块设计.md) 基础上，依据《湛江市国资国企一体化数字平台功能清单（二期）.xlsx》的60项功能清单进行了全面补充和细化。

**主要变更：**
- 模块从 5个业务服务 扩展为 **16个功能模块、60+功能点**
- 新增：巡检管理、盘点管理、采购管理、维保维修管理、预警中心
- 补充：资产基础资料（位置/供应商/维保商）、资产标签、操作日志
- 审批工作流从笼统描述升级为可配置审批引擎
- 交付周期从 7周 调整为 **14周两阶段交付**

## 二、关键设计决策

| # | 决策项 | 选择 | 说明 |
|---|--------|------|------|
| 1 | 覆盖范围 | **功能清单16模块全部纳入** | 企业内资产管理 + 国资监管上报全覆盖 |
| 2 | 交付周期 | **14周两阶段（8-9月）** | Phase 1(7月底)：台账+流转+采购+折旧+驾驶舱；Phase 2(8-9月)：巡检+盘点+维保+审批+预警+监管上报 |
| 3 | 系统架构 | **模块化单体 + 三域划分** | 企业资产管理域 / 国资监管域 / 平台基础域，域间通过Service接口通信 |
| 4 | 企业层vs监管层 | **平行独立、数据共享** | 企业域数据作为监管上报数据源，通过接口松耦合 |
| 5 | 审批工作流 | **可配置审批引擎** | 按业务类型定义审批链，支持条件分支（如金额阈值） |
| 6 | 资产管理深度 | **全生命周期** | 从采购申请到报废退出的完整闭环 |
| 7 | 数据采集策略 | **混合模式** | Excel批量导入 + 财务系统对接 + 人工录入 |
| 8 | 部署模式 | **SaaS多租户** | 统一部署，tenant_id数据隔离 |
| 9 | 技术栈 | **Java/SpringBoot全栈** | SpringBoot 3 + Vue3 + Element Plus + PostgreSQL + MyBatis-Plus |

### 3.2 域间关系

- **企业域 → 监管域**：企业资产台账通过数据接口提供给监管上报服务，生成19类报表快照。企业域不感知监管上报的存在，松耦合。
- **审批工作流跨域共用**：审批引擎作为独立模块，服务于调拨审批（企业域）、盘点差异审批（企业域）、上报审核（监管域）。
- **平台域横切能力**：多租户隔离、RBAC权限、操作日志、数据导入导出由平台域统一提供，其他域自动继承。

## 五、审批工作流设计

### 5.1 核心模型

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| ApprovalDef | 审批定义模板 | biz_type（调拨/报废/盘点差异/采购/维保/上报）, trigger_condition(JSON), status |
| ApprovalNode | 审批节点 | node_order, approver_role, condition_expr（可选条件表达式）, can_reject, timeout_hours |
| ApprovalInstance | 审批实例 | biz_id, def_id, current_node, status(PENDING/APPROVED/REJECTED/CANCELLED), node_results(JSONB) |

### 5.2 支持的审批业务类型

| 业务类型 | 默认审批链 | 条件分支 |
|---------|-----------|---------|
| 资产调拨 | 调入部门确认 | — |
| 资产报废 | 部门负责人 → 财务审核 | 原值>10万：追加分管领导 |
| 盘点差异 | 差异确认 → 财务调整 | — |
| 采购申请 | 部门负责人 → 管理层 | 金额>5万：追加财务审核 |
| 维保维修 | 申请 → 部门确认 | — |
| 监管上报 | 企业提交 → 集团审核 → 国资委接收 | — |

### 5.3 审批动作

- **通过**：流转到下一节点，或标记实例为 APPROVED
- **驳回**：退回到发起人，发起人可修改后重新提交
- **退回上一节点**：退回当前节点的上一级重新审批
- **加签/转审**（二期扩展）：将审批转给其他人

## 六、核心业务流程

### 6.1 巡检管理流程

```
制定巡检任务 → 任务分发 → 现场巡检登记 → 异常处理 → 任务完成归档
（选择资产范围、 （巡检人收到       （逐项检查、        （正常→完成、     （统计报告、
 指定巡检人、     "我的巡检任务"、   记录状态/位置、     异常→转维修/      归档、
 设定截止时间）    系统跟踪进度）     拍照留证）         安排整改）         超期预警）
```

### 6.2 盘点管理流程

```
制定盘点计划 → 现场盘点 → 系统自动差异识别 → 差异处理 → 盘点归档
（按部门/位置/    （盘点人逐项核实、   （账面vs实际比对、    （差异报告、     （盘点报告、
 资产范围生成     记录实际数量/       区分盘盈/盘亏/       审批确认、      差异存档、
 盘点清单）       位置/状态）         位置不符/状态不符）   台账调整）      审计备查）
```

**关键点：** 差异识别由系统自动完成，人工确认后通过审批流程调整台账。

### 6.3 采购管理流程

```
采购申请 → 审批 → 采购验收 → 采购转资产
（填写需求、     （部门→管理层、    （到货核对、        （自动带入采购信息、
 关联供应商、     金额条件分支、     数量/规格/质量、    生成资产编号、
 预算金额）       审批工作流）       验收结果登记）      入库登记）
```

**关键点：** 验收通过后"采购转资产"一步完成资产登记，避免重复录入。

### 6.4 维保维修管理（双路径）

- **路径A（手动发起）：** 维保维修申请 → 关联维保商 → 派工处理 → 维修登记（结果/费用） → 维修历史归档
- **路径B（巡检转入）：** 巡检异常 → 一键转维修 → 自动带入资产信息+异常描述 → 派工处理 → 维修登记 → 异常闭环

### 6.5 预警中心 — 4类自动预警

| 预警类型 | 触发规则 | 扫描逻辑 |
|---------|---------|---------|
| 维保到期预警 | 维保到期日前7天/当天/超期触发三级预警 | 定时扫描 MaintenanceRecord 维保到期日 |
| 巡检超期预警 | 超过计划截止日期仍未完成 | 扫描 InspectionTask 中 end_date 已过且 status≠已完成 |
| 借用超期预警 | 超过约定归还时间 | 扫描 Assignment 中借用类型，expected_return_date 已过 |
| 闲置资产预警 | 使用状态为IDLE且超过N天无变动 | 扫描 use_status=IDLE 且 updated_at 超过N天的资产 |

预警统一使用 AlertRule 配置阈值参数，定时任务（Spring @Scheduled）批量扫描，生成 AlertRecord 推送到工作台。

### 6.6 现有流程（保持v1设计）

- **资产全生命周期：** 资产新增 → 使用管理 → 变动流转 → 折旧摊销 → 资产处置
- **数据上报-确认模式：** 企业填报 → 集团审核 → 国资委接收 → 省平台对接
- **财务系统对接：** 企业财务系统（用友/金蝶/SAP）→ 财务适配器层 → 数据校验引擎 → 平台资产台账

## 七、多租户与权限模型

与 v1 设计保持一致，补充：

### 7.1 数据隔离

| 维度 | 方案 |
|------|------|
| 数据库级 | 共享数据库 + tenant_id 字段隔离 |

| 企业操作员 | 管理本企业资产、填报数据、提交上报 |
| 系统管理员 | 租户管理、用户管理、配置字典、系统运维 |

## 九、部署架构（MVP）

## 十三、错误处理策略

| 错误类型 | HTTP状态码 | 处理方式 |
|---------|-----------|---------|
| 参数校验失败 | 400 | GlobalExceptionHandler 统一拦截 MethodArgumentNotValidException |
| 业务逻辑错误 | 400 | BusinessException，返回用户可读的中文错误信息 |
| 认证失败 | 401 | Spring Security 拦截，返回"登录已过期" |
| 权限不足 | 403 | 返回"无权限访问" |
| 资源不存在 | 404 | BusinessException，返回"XX不存在" |
| 系统内部错误 | 500 | 全局兜底，记录详细日志，返回"系统内部错误" |

## 十五、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 14周范围大，交付压力 | 质量不达标 | 分期交付，Phase 1聚焦核心闭环；Phase 2可剪裁非核心预警类型 |
| 审批引擎可配置复杂度高 | Phase 1 延期 | Sprint 2 先做固定链审批（硬编码），Sprint 3 再抽象为可配置引擎 |
| 财务系统对接难度高 | 数据拿不到 | 适配器层设计灵活，Excel模板兜底，Phase 2 先对接1家试点 |
| 新模块多，测试覆盖不足 | Bug多 | TDD强制，每个Sprint做测试覆盖率检查 |
| 企业配合度不足 | 数据进不来 | Excel批量导入降低门槛，提供驻场培训支持 |
| 信创环境兼容 | 部署受阻 | 技术栈全部基于信创兼容选型，MVP可用标准环境先跑 |
